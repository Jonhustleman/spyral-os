# SPYRAL OS — Performance Budgets

**Last Updated:** 2026-07-20  
**Status:** Active  
**Owner:** Implementation Engineer  

## Purpose

Performance budgets protect user momentum. They establish clear thresholds that, when exceeded, trigger investigation before the change reaches production.

## Budgets

### Technical Budgets (Build-Time)

| Category | Metric | Budget | Warning At | Measurement |
|----------|--------|--------|------------|-------------|
| JavaScript (total) | Total JS bundle (gzip) | < 200 KB | 180 KB | `@next/bundle-analyzer` |
| JavaScript (initial) | Initial page JS (gzip) | < 100 KB | 90 KB | `@next/bundle-analyzer` |
| CSS | CSS bundle (gzip) | < 50 KB | 45 KB | `@next/bundle-analyzer` |
| Images | Total image weight per page | < 500 KB | 400 KB | Manual review |
| Total page weight | Full page (HTML + JS + CSS + images) | < 800 KB | 700 KB | Lighthouse |

### Interaction Budgets (Runtime)

| Metric | Budget | Warning At | Measurement |
|--------|--------|------------|-------------|
| First Contentful Paint (FCP) | < 1.5s | 1.2s | `performance.timing` / PaintTiming API |
| Largest Contentful Paint (LCP) | < 2.5s | 2.0s | PerformanceObserver |
| Interaction to Next Paint (INP) | < 200ms | 150ms | PerformanceObserver |
| Navigation transition | < 150ms | 100ms | Store subscription timestamps |
| Studio render time | < 100ms | 80ms | `useEffect` + `performance.now()` |
| localStorage read | < 50ms | 30ms | Wrapper timing in storage adapter |
| localStorage write | < 50ms | 30ms | Wrapper timing in storage adapter |

### Product Budgets (Cognitive)

| Metric | Budget | Notes |
|--------|--------|-------|
| Time to First Insight | < 30s | How long before the user gains new understanding |
| Average Journey Completion | < 5 min | Full intent → learning cycle |
| Questions per Journey | 3-7 | Too few = shallow, too many = friction |
| Resume Rate | > 80% | Users who return within 24h to continue |
| Validation Completion Rate | > 90% | Users who finish validation once started |
| Learning Generation Rate | > 70% | Journeys that produce at least one insight |

## Enforcement

### CI Enforcement

- **Bundle size warnings** are generated by `npm run analyze` (uses `@next/bundle-analyzer`)
- **Bundle reports** are uploaded as CI artifacts for PR review
- Performance budget violations should be flagged during code review

### Runtime Enforcement

- Performance metrics are collected by `src/lib/performance.ts`
- Metrics are logged through the observability layer (ADR-0059)
- Developer Mode (`src/components/dev/DeveloperMode.tsx`) displays live metrics
- `/debug` route (dev-only) shows historical performance data

## Workflow

1. Developer makes a change
2. CI runs `npm run analyze` — generates bundle report
3. If bundle exceeds warning thresholds, developer investigates
4. CI uploads bundle report as artifact for reviewer
5. Reviewer checks for regressions before approving

## Exceptions

Performance budget exceptions require documented justification in the PR description explaining:
- Why the increase is necessary
- What user-facing improvement it enables
- Plan to optimize in a follow-up
